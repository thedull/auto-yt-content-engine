## What this video covers

In this video, you are going to learn everything you need to know about testing your React applications using **Playwright**. Playwright is an **end-to-end testing tool** that literally allows you to test your applications exactly as a real user would. It does that by spinning up a browser, running your application, and going and clicking on things, typing on inputs, and making sure that your application behaves in its entirety — frontend, backend, and database included — exactly as you expect, and that there's no unwanted surprises.

## End-to-end vs unit testing

To get started we go to the Playwright documentation at **playwright.dev**. In this video I'm going to show you how we could test this very simple application, but more importantly I'm also going to walk you through my **thought process of what to test**, because unfortunately a lot of tutorials don't actually show this — they show you how to use the tool and how to write tests, but they never teach you the core principles of testing and *why* you should test something in the first place, and more specifically what to test versus what not to test.

As the docs say, *Playwright enables reliable end-to-end testing for modern web applications.* If you don't know what end-to-end testing is, as opposed to **unit testing**: end-to-end testing means that you're taking the entire application and you're actually running it with the whole front end, the whole back end, and a database, and you're testing it as a real user would. You go and click on things, type on inputs, and see how the application behaves — as opposed to a unit test, where you're testing a component or something in isolation.

## Deciding what to test

Let me run the application again with `pnpm dev`. Let's think about what we want to test.

**Homepage.** The home screen only has a few elements, so there's not really that much to test. But we should still test that what we're seeing on the screen is exactly what we expect:

- A heading on the page that says "homepage" — with the correct text.
- A link that says "form" is there, and that clicking it successfully redirects to `/form`.
- The page title is "Cosden Solutions", just for good measure.

**Form page.** Has a little bit more functionality:

- The form is actually there.
- The input and the **Add** button are there.
- I can type something in the input.
- When I type and click Add, whatever I typed ends up in a list and the input is cleared.

Note that this doesn't depend on any individual piece — not just the front end, not just the back end, not just the database — it depends on all the pieces working together. If we had a back end that saved everything I type, we would also be indirectly testing that, because the only way the item shows up here is if the back end is also working. This is what we call an **integration test**. In our simple example everything is handled on the front end, but you can imagine if we had a back end, this test would cover it.

That's generally how you want to approach thinking about testing and what to test: take any screen, look at what any user can do, what they expect to see, and just test those paths and make sure they're correct. Also test that whenever there's an error, your application responds in a way that you expect.

## Setting up a `describe` block for the homepage

I'm going to do all of this work in one `example.spec.ts` file just to keep things simple. In a real application you might want to split this up into multiple files, each related to different types of tests — that's totally recommended.

We're going to have two different groups of tests because we have two different pages: the homepage and the form page. It's generally a better idea to group related tests together. In Playwright you do that by using a **`describe` block**:

```ts
test.describe('home page', () => {
  // tests go here
})
```

The only thing this `describe` block actually does is, when you see the tests being run in the UI, they're going to be grouped under this category to make it easier for yourself to organize. I always like to do this — group related tests together because it just makes more sense.

## First test: page title

The first test we want to do is open up `localhost:3000` and check the title.

```ts
test('should have correct metadata and elements', async ({ page }) => {
  await expect(page).toHaveTitle('Cosden Solutions')
})
```

You always want to give the test a **descriptive title** to actually represent what the test is doing. The async function takes an object from which we destructure `page` — we're going to use this `page` to access different things on that specific page, like the title and any elements, and check that they're exactly what we expect.

`expect` comes directly from `@playwright/test`, the import at the top.

But before we can run this there's one more thing we have to do that I left out on purpose just to make it really obvious how important this is. **You cannot expect the page to have anything before even visiting the page in the first place.** Right now we're never telling the test to navigate anywhere. We have to navigate first:

```ts
await page.goto('http://localhost:3000')
await expect(page).toHaveTitle('Cosden Solutions')
```

You have to give this to each test, because each test runs in isolation and needs to know what page to actually go to.

## Filling out the homepage test: heading and link

We don't only want to test that the page has the correct title. We also want to test that we have a heading that says "homepage" and a link to the form page.

```ts
await expect(page.getByRole('heading', { name: 'homepage' })).toBeVisible()
```

`getByRole` gets an element by its **role**, in this case `'heading'`, and we can give properties like `name`, which is the text inside that heading. If the heading didn't have "homepage", we wouldn't get this element. We then call `toBeVisible()` to check the element is actually visible on the screen.

Let's press play in the UI again — it passes. You can see it highlighting the homepage heading.

Just to be sure our test is actually working, I like to do this sometimes — you never know. Let me add an extra `e` to make the name "homepageee", an invalid name. Now the test should fail. When it fails it actually retries the test, so it takes a little longer, but soon we see *expect to be visible* failed because there's no heading with "homepageee". Now we've confirmed our test is actually behaving as we expect and we can rely on it to fail when it should.

Let's fix that and add the link test:

```ts
await expect(page.getByRole('link', { name: 'form' })).toBeVisible()
```

Rerun the tests — all green. We've successfully completed our first actual test, *should have correct metadata and elements*.

## Second test: redirect on link click

While it's cool we tested that the link is there and says "form", we did not test that clicking it brings us to the expected page. This is also important because the user is going to click on it, and if they're not redirected to the form, you have a bug. Having a test for that prevents the bug.

```ts
test('should redirect to form page on click', async ({ page }) => {
  await page.goto('http://localhost:3000')
  await page.getByRole('link', { name: 'form' }).click()
  await expect(page).toHaveTitle('form')
})
```

We're going to have to write the same `page.goto` again because **every single test in Playwright runs in isolation**. You have a whole new session every time, which means a fresh browser and the same page. So we have to write this for every test.

But doing it this way is not the recommended way because you're repeating yourself. On the homepage, in this `describe` block, most if not all of our tests are going to need to go to the same page. It would be better if we had a way to run something **before every single test** where we can put this little piece of code.

## `beforeEach` hooks

The cool thing about Playwright is there is. We can inject a hook **before every single test**. We do that inside the `describe` block, because we only want this hook to apply to tests inside that block — that's another use case for `describe`:

```ts
test.beforeEach(async ({ page }) => {
  await page.goto('http://localhost:3000')
})
```

Now we don't need to have `page.goto` inside each individual test. Just to prove this, I'll comment out that line in the first test and run it — it still passes, because the hook runs before each test.

Now let's finish the redirect test. Instead of `toBeVisible` we call `.click()`, then await `expect(page).toHaveTitle('form')` — whenever we go to the form page the title actually changes to "form". We could also check the URL, but we'll keep things simple.

Run all the tests in the UI — both pass. The second one shows a little **red circle** where it clicked the form element, and then it checked the actual page: we're on `/form` and the title is "form".

With this, our tests are passing and we can guarantee our application is going to work exactly as we expect now. And even more importantly, **if we ever make a change to any code in the future**, having these tests is going to make sure that if we made some change that broke something, one of these tests is going to break and we're going to know about it. That's why you want tests in your application.

Having a visual runner where you can inspect every test, see the UI rendered at each step, and see what the test runner is trying to click is really, really useful for debugging. Whenever you're developing your tests I'd recommend you use this UI.

## The form-page `describe` block

Now we have to write tests for the form page. I'll go a bit faster here because I don't want to drag the video on by repeating the same things.

Create a new `describe` block for the form page, and a `beforeEach` hook that navigates to `localhost:3000/form`:

```ts
test.describe('form page', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('http://localhost:3000/form')
  })

  // tests...
})
```

The first test is literally the same as before — should have correct metadata and elements — but for the form page:

- Title is "form"
- Heading text is "form"
- Instead of a link we have an input — get it by **placeholder**:

In the form component, the input has a placeholder that says "Enter item". Because this is the only element with this placeholder, it's safe to use:

```ts
await expect(page.getByPlaceholder('Enter item')).toBeVisible()
```

`getByPlaceholder` only takes one argument, so we don't pass a second.

We also have a button that says "Add":

```ts
await expect(page.getByRole('button', { name: 'Add' })).toBeVisible()
```

Run it — we now have a new `describe` section in the UI runner (one of the benefits of `describe` blocks), and the test passes.

## Empty initial state with `data-testid`

There are two more tests I want to write:

1. Initially, when we load the page, we should have **no items displayed** — our initial empty state.
2. Once we enter an item, we should see that item in the list and the input should be cleared.

To do this we need to add some properties to the form component so we can reference elements in our test. We're going to use the **`data-testid` attribute**. This is an attribute you can place on any element — just like a normal prop — give it a specific test ID, and then use that test ID to reference the element from your test.

On the unordered list:

```jsx
<ul data-testid="items-list">
```

Now the empty test:

```ts
test('should have empty items list on start', async ({ page }) => {
  const itemsList = page.getByTestId('items-list')
  await expect(itemsList).toBeEmpty()
})
```

We could have done this inline without the variable — that would have worked — I'm just showing you a different way, because we're working in JavaScript and TypeScript and we can create variables. Sometimes it's useful if you want to **reuse** the locator in other places.

Run it — it gets the element by its test ID, sees it's empty, test passes.

## Adding an item to the list

The last test checks that when we enter an item and click Add, the item is added and the input is cleared.

In the form component, on each `<li>` we add a test ID:

```jsx
<li data-testid="item">
```

The test:

```ts
test('should add item to list', async ({ page }) => {
  const input = page.getByPlaceholder('Enter item')
  await input.fill('item 1')

  await page.getByRole('button', { name: 'Add' }).click()

  const item = page.getByTestId('item').nth(0)
  await expect(item).toHaveText('item 1')
  await expect(input).toBeEmpty()
})
```

A couple of things to note:

- **`input.fill('item 1')`** sets a value on the input field. This is what you use whenever you want to put text into a specific input.
- After clicking Add, we need to get the new item. Because the list maps over items each `<li>` has the same `data-testid="item"`, so we use **`.nth(0)`** to access the first item.
- We assert the item has text "item 1" and the input is empty (because every time we click Add the input should clear so we can type something else).

Replay all the tests in the runner and every single test passes. *Should add item to list* passes — we fill the text, you can see "item 1" appear in the input, we click the button, the item appears, we check it has "item 1", and we check the input is actually empty.
